QR1. Which concept was hardest to understand: lists and loops, classes and methods, exception handling, or async/await? Walk through the specific moment of confusion and explain how you eventually understood it.
The concept that was hardest for me to understand was async/await. At first, it was confusing because the program did not run in the same top-to-bottom order as normal code. When I saw Future and await, I did not fully understand why the program needed to “wait” or what was happening behind the scenes.
The moment of confusion happened when I saw Future.delayed() in Task 2. I expected the program to run immediately, but instead there was a pause before the result was printed. I realized that the function was returning a Future, which represents a value that will be available later.
I eventually understood it by thinking of asynchronous programming as starting a task that takes time, like downloading data from the internet. The program continues running while waiting for that task to finish. When I used await, it told the program to pause at that point until the result was ready. After practicing with small examples and seeing how the delay worked in the calculator program, the concept became clearer.

QR2. Look at your Task 1 solution. If you had to change your code to analyze a list of floating-point numbers (List<double>) instead of integers, how many places in your code would need to change? What does this tell you about the importance of thinking about data types during design?
If the program needed to analyze a list of floating-point numbers instead of integers, several parts of the code would need to change. The list type would need to change from List<int> to List<double>. In addition, variables used inside the functions such as max, min, and sum would also need to be changed from int to double.
Functions that return values, such as findMax(), findMin(), and calculateSum(), would also need to return double instead of int. The list declared in the main() function would also need to use decimal numbers instead of integers.
Overall, multiple locations in the program would need to be modified. This shows that choosing the correct data type during the design stage is very important. If the wrong data type is used, changing it later can require many modifications across the program. Thinking carefully about data types early in development helps make the code more flexible and easier to maintain.

QR3. In Task 2, you simulated a network delay using Future.delayed(). Describe a real Flutter app feature where you would need to actually await a real asynchronous operation — not just a simulated delay. What would the Future resolve to, and what would happen to the UI while the user was waiting?
A real example of asynchronous operations in a Flutter app is loading data from an online API, such as retrieving a list of products in a shopping app or loading posts in a social media feed.
In this case, the app would send a request to a remote server to fetch data. The operation would return a Future, which would eventually resolve to the data received from the server, such as a list of objects or JSON data that represents the items to display in the app.
While the user is waiting for the response, the UI should not freeze. Instead, the app would display a loading indicator or progress spinner to inform the user that data is being loaded. Once the Future completes and the data is received, the UI would update to display the content.
This asynchronous behavior allows the application to remain responsive while waiting for operations like network requests, file reading, or database queries to finish.

QR4. If a colleague asked you: “Why not just use a single function that takes the operation name as a parameter, rather than writing separate add(), subtract(), multiply(), and divide() methods?” — what would you say? Are there advantages to both approaches?
Using a single function that accepts the operation name as a parameter can make the code shorter and reduce the number of methods. In that approach, the function could use a switch statement to determine which operation to perform. This can be convenient for simple programs.
However, writing separate methods such as add(), subtract(), multiply(), and divide() has several advantages. Each function has a clear responsibility, which improves readability and makes the code easier to understand. It also follows good software design practices such as the Single Responsibility Principle, where each function performs one specific task.
Separate methods also make the program easier to test and maintain. If there is a bug in the multiplication logic, for example, it can be fixed directly in the multiply() method without affecting other operations.
Both approaches have advantages. A single function may be simpler for very small programs, but using separate methods usually leads to cleaner, more modular, and more maintainable code, especially as the program grows larger.
